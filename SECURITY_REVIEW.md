# FocusFriends — обзор безопасности и внесённые исправления

Дата: 2026-07-22

## Как искали

Клонирован репозиторий, вручную прочитан код всех Java-сервисов
(`auth-service`, `gateway-service`, `core-service`, `social-service`,
`notification-service`, `health-service`), `docker-compose.yml` и
конфигурация Spring Security/Spring Cloud Gateway. Основное внимание —
модель доверия между gateway и внутренними сервисами, т.к. в
микросервисной архитектуре именно там чаще всего прячутся дыры в
контроле доступа.

## Найденные уязвимости

### 1. Критично: подмена личности (X-User-Id spoofing) — полный обход аутентификации

`core-service`, `social-service`, `notification-service`, `health-service`
**слепо доверяли** заголовку `X-User-Id`, считая, что его мог поставить
только gateway после проверки JWT. Сам заголовок никак не подписан и
ничем не защищён.

При этом `docker-compose.yml` публиковал порты **всех** сервисов на хост
(`8081`–`8085`), то есть любой, кто может достучаться до контейнера
(например, если хост доступен из интернета/локальной сети), мог
обратиться к сервисам напрямую, минуя gateway, и просто подставить
`X-User-Id: <любой чужой UUID>` — получив полный доступ к задачам,
привычкам, друзьям, группам и данным здоровья любого пользователя без
единого пароля или токена.

**Исправление:**
- Добавлен общий секрет (`X-Internal-Secret`, конфигурируется через
  `INTERNAL_SHARED_SECRET`), который gateway проставляет в каждый
  запрос, прошедший проверку JWT/API-токена (`JwtAuthenticationGlobalFilter`).
- Новый `InternalSecretInterceptor` в каждом из 4 сервисов отклоняет
  (403) любой запрос без верного секрета — константным по времени
  сравнением (`MessageDigest.isEqual`), до какой-либо бизнес-логики.
- Gateway теперь явно **удаляет** заголовки `X-User-Id` и
  `X-Internal-Secret`, присланные клиентом, прежде чем проставить
  настоящие — иначе клиент мог бы попытаться подсунуть свой секрет.
- `docker-compose.yml`: порты `auth-service`, `core-service`,
  `social-service`, `notification-service`, `health-service` больше не
  публикуются на хост (`expose` вместо `ports`) — снаружи доступны
  только `gateway-service` (8080) и `frontend` (3000).

### 2. Критично: обход контроля доступа на «внутренних» эндпоинтах через сам gateway

`POST /api/leaderboard/points` (social-service) и `POST
/api/notifications/send` (notification-service) задумывались как
эндпоинты только для вызовов сервис-сервис (это подтверждают
комментарии в коде), поэтому были **исключены** из проверки
`X-User-Id` в конфигурации соответствующих сервисов.

Но маршруты gateway (`/api/leaderboard/**`, `/api/notifications/**`)
захватывали их тоже, а сам gateway не считал эти пути публичными —
значит, **любой залогиненный пользователь** мог обратиться к ним через
gateway со своим обычным JWT и:
- начислить себе (или кому угодно) произвольное количество очков в
  лидерборде (`awardPoints` принимает произвольный `userId` и
  `points`);
- отправить произвольное уведомление/письмо любому пользователю от
  имени системы (`send` не проверяет отправителя) — потенциальный
  вектор для спама/фишинга.

**Исправление:**
- В `JwtAuthenticationGlobalFilter` добавлен явный список
  `service-only-paths` (`/api/leaderboard/points`,
  `/api/notifications/send`): запросы к этим путям **всегда**
  отклоняются с 404, даже с валидным JWT — снаружи они не существуют.
- Одновременно они, как и всё остальное, защищены секретом из пункта 1
  — это защита в глубину на случай прямого доступа к порту сервиса.
- Клиенты, которые реально должны вызывать эти эндпоинты
  сервис-сервис напрямую (`NotificationServiceClient` в
  `auth-service`/`social-service`), обновлены — теперь отправляют
  `X-Internal-Secret`.

## Более мелкие наблюдения (не исправлялись, стоит учесть)

- **JWT-секрет по умолчанию.** И в `auth-service`, и в
  `gateway-service` есть dev-фолбэк для `JWT_SECRET`
  (`this-is-a-dev-only-secret-key...`). Если переменную окружения
  забудут выставить в проде — токены можно подделать. Рекомендация:
  на проде явно требовать секрет (fail-fast, если переменная не
  задана), а не подставлять дефолт.
- **Нет rate limiting на `/api/auth/login` и `/api/auth/register`.**
  Ничего не мешает перебору паролей/e-mail enumeration через простой
  brute force. Стоит добавить ограничение частоты запросов (на
  gateway или через что-то вроде Bucket4j/Redis).
- Хорошие практики, которые уже были: пароли хранятся через BCrypt,
  refresh-токены хранятся хэшированными/отзываемыми в БД, персональные
  API-токены (`ff_pat_...`) хранятся только как SHA-256 хэш,
  `requestPasswordReset` не даёт понять, существует ли e-mail
  (защита от user enumeration), публичный профиль пользователя отдаёт
  отдельный урезанный DTO без e-mail/пароля.

## Тесты

Добавлены юнит-тесты, которые падали бы на старом коде и проходят на
исправленном:

- `gateway-service/.../JwtAuthenticationGlobalFilterTest.java` —
  проверяет, что: (a) заголовок `X-User-Id`, присланный клиентом,
  игнорируется и заменяется настоящим; в forwarded-запрос
  проставляется `X-Internal-Secret`; (b) невалидный токен → 401;
  (c) `/api/leaderboard/points` и `/api/notifications/send`
  недостижимы через gateway даже с валидным JWT (404); (d) публичные
  пути по-прежнему работают без токена.
- `core-service`, `social-service`, `notification-service`,
  `health-service`: `InternalSecretInterceptorTest.java` — без
  секрета/с неверным секретом → 403, с верным — пропускает запрос.

## Как проверить локально

```bash
cd auth-service && mvn test
cd ../gateway-service && mvn test
cd ../core-service && mvn test
cd ../social-service && mvn test
cd ../notification-service && mvn test
cd ../health-service && mvn test
```

> В песочнице, где выполнялась эта проверка, нет доступа к Maven
> Central (сеть ограничена npm/PyPI/GitHub), поэтому `mvn test`
> здесь не запускался — код проверен вручную построчно, но
> рекомендуется прогнать тесты локально/в CI перед деплоем.
